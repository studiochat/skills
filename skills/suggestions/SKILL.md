# Suggestions API

A **suggestion** is an informational improvement hint attached to a project (the
assistant). It is **not directly actionable** — a human reviews it in the
Suggestions inbox and marks it **resolved** (or reopens it). Suggestions are the
input to the [continuous-improvement](../continuous-improvement/SKILL.md) loop:
this skill records *what to improve*; continuous-improvement *ships* the change.

All API calls are authenticated via environment variables and go through the
`scripts/fetch.py` helper (same pattern as [data-expert](../data-expert/SKILL.md)).
The API base URL (`https://api.studiochat.io`) is the default.

## Setup

```bash
export STUDIO_API_TOKEN="sbs_your_project_sandbox_key"   # account-scoped key
export STUDIO_PROJECT_ID="your-project-uuid"
```

The sandbox key (`sbs_`) is account-scoped, so it can create/read suggestions
for any project in its account. (A cross-account operator agent can instead use
a fine-grained `agt_` token with the `suggestions:read` / `suggestions:write`
scopes against the `/supersami/projects/{id}/suggestions...` mirror — same
shapes, different prefix.)

## The data model

A suggestion has:

| field | notes |
|---|---|
| `id` | **incremental integer** (global). This is what you reference to add follow-ups. |
| `title` | short summary (shown in the list) |
| `description` | one-line development (optional) |
| `body` | full free-text "smart text" — markdown + inline `[[ref ...]]` macros (see below) |
| `category` | closed enum, currently only `kb` (knowledge-base improvements). Defaults to `kb`. |
| `status` | `open` or `resolved` (a human flips this). New suggestions are `open`. |
| `occurrence_count` | **derived** = 1 + number of follow-ups |
| `last_seen_at` | bumped each time a follow-up is added |
| `followups` | timeline of later detections (only populated on the detail endpoint) |

### Body macros (`[[ref ...]]`)

The `body` is rendered in the UI as rich text. Embed references to resources
inline and they render as pills — **always reference by id**:

```
[[ref type="kb" id="<kb_uuid>" label="Pagos"]]
[[ref type="conversation" id="<conversation_id>" label="convo 1"]]
[[ref type="skill" baseId="<assistant_id>" label="Reembolsos"]]
[[ref type="instructions" baseId="<assistant_id>" label="Instrucciones"]]
[[ref type="tool" label="crear_ticket"]]
```

- `kb` resolves strictly by `id` to the KB's title; if the id doesn't exist the
  UI shows "KB not found" — never invent an id.
- `conversation` links to the chatlog by its `conversation_id`.
- Use these to cite *where* the gap was seen (the conversations) and *what*
  resource it's about (the KB).

## Grouping & follow-ups (read this)

There is **no dedup key / no semantic matching**. Every `create` inserts a new
suggestion. To avoid duplicates for the same problem, the agent itself decides
grouping:

1. **List** the project's existing suggestions first.
2. If the thing you just detected is the **same problem** as an existing
   suggestion, add a **follow-up** to that suggestion's `id` (this reopens it if
   it was resolved — a regression — and bumps `last_seen_at` / `occurrence_count`).
3. Otherwise, **create** a new one.

So a typical hourly run is: *list → for each finding, match against existing →
follow-up or create*.

## Endpoints

Use `python3 scripts/fetch.py <path> [--method POST] [--body '{...}'] [--params k=v]`.

### List (do this first, to decide create vs follow-up)
```bash
python3 scripts/fetch.py "/projects/$STUDIO_PROJECT_ID/suggestions" \
  --params status=open category=kb limit=100
```
Returns `{ items: [...], total, open_count }`. Items are light (no `followups`).
Filters: `status` (`open`/`resolved`), `category` (`kb`), `limit`, `offset`.

### Get one (full detail, with follow-up timeline)
```bash
python3 scripts/fetch.py "/suggestions/<id>"
```

### Create
```bash
python3 scripts/fetch.py "/projects/$STUDIO_PROJECT_ID/suggestions" \
  --method POST --body '{
    "title": "Falta info de horarios en la KB",
    "description": "Varios usuarios preguntaron y el bot no supo responder.",
    "body": "En la base [[ref type=\"kb\" id=\"KB_UUID\" label=\"Contacto\"]] falta el horario. Visto en [[ref type=\"conversation\" id=\"CONV_ID\" label=\"convo 1\"]].",
    "category": "kb"
  }'
```
Returns the created suggestion (note its integer `id`).

### Add a follow-up (same problem detected again)
```bash
python3 scripts/fetch.py "/suggestions/<id>/followups" \
  --method POST --body '{ "body": "Siguen entrando casos: [[ref type=\"conversation\" id=\"CONV_ID_2\" label=\"convo 2\"]]." }'
```
Reopens the suggestion if resolved; `occurrence_count` goes up.

### Resolve / reopen (usually a human does this in the UI)
```bash
python3 scripts/fetch.py "/suggestions/<id>/resolve" --method POST
python3 scripts/fetch.py "/suggestions/<id>/reopen"  --method POST
```

## Conventions for good suggestions

- **One problem per suggestion.** Don't bundle unrelated gaps.
- **Title**: short and specific ("Falta info de horarios en la KB"), not generic.
- **Body**: explain the gap, cite the conversations where it showed up (`[[ref
  type="conversation" ...]]`) and the KB it concerns (`[[ref type="kb" id=...]]`),
  and propose the fix. Keep it concrete.
- **Category** is `kb` for now (this is the only kind we generate).
- **Always list first** and prefer a follow-up over a near-duplicate create.
